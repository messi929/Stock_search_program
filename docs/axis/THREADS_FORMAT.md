# 스레드 전용 포맷 재설계 — 논의 기록

> **상태**: **방향 확정(2026-07-14).** 착수 순서 = 타래 인프라 → 프롬프트/루브릭 → 이미지.
> **계기**: "우리 마케팅글이 스레드와 어울리지 않는다"는 지적.
> 관련: `docs/axis/MARKETING.md`(현행 공장), `agents/marketer.py`, `utils/threads_client.py`

---

## 1. 결론 요약

**문제는 길이도, 해시태그도, 서식도 아니다.** 그건 이미 맞춰져 있다(500자 상한, 해시태그 제거 완료, 줄바꿈 호흡).

진짜 문제는 두 가지다.

1. **콘텐츠 단위** — 우리는 "종목 1개 + 수치 나열 + 관찰포인트 번호 + 링크"를 쓴다. 이 형태는 스레드에서 구조적으로 죽는다. (아래 §2 증거)
2. **화자 부재** — 데이터는 있는데 말하는 사람이 없다. 스레드는 짧은 글의 공간이 아니라 **화자가 있는 글의 공간**이다.

그리고 우리가 **손에 쥐고도 안 쓰던 무기**가 하나 있다 — 스레드 주식판은 지금 "추천글 홍수"에 대한 불신이 폭발 직전이고, Axis는 애초에 추천을 하지 않는 제품이다.

## 2. 증거 (1차 자료)

### 2-1. 같은 계정 안의 대조 — `@stockingresearcher` ("주식하는 연구원")

| 글 | 반응 |
|---|---|
| 개별 종목 분석 (파가야, 컵앤핸들 패턴, **차트 이미지 첨부**) | 좋아요 **3**, 댓글 1 |
| 거시 서사 (관세 → 재고 → 물가 → 금리인하 → 아마존 시나리오) | 좋아요 **231**, 댓글 27, 리포스트 15 |

같은 사람·같은 톤·같은 계정. 차이는 **"종목 데이터 리포트냐, 세상에 대한 이야기냐"** 하나뿐.
→ **우리 현행 종목글은 좋아요 3을 받은 쪽과 구조가 동일하다.**

### 2-2. `@us_stock_info` (미국주식가이드)
- 최고 반응(좋아요 121): 급락장에서의 **개인 심리 경험담**
- 톤: 반말/구어체, ㅋㅋ·ㅠㅠ

### 2-3. 실제 대박 타래 (2026-07-13, `@pro_benefit_`)
코스피 7000선 붕괴 당일. **조회 87,585 / 좋아요 283 / 댓글 14 / 리포스트 8**

구조:
```
[루트] 🚨와... 코스피 7000선 붕괴 ㄷㄷ
       오늘 시장에서 가장 무서운 건 하락이 아니라,
       '왜 떨어지는지' 모른 채 버티는
       투자자들이 늘고 있다는 점임. (1/10)
       + 뉴스 기사 링크 카드(이미지)
[2/10] 1/ 현재 코스피는 6941까지 밀렸음. …
[3/10] 2/ …
  … 총 10파트 타래
```

**핵심 관찰:**
- 뉴스는 **훅**일 뿐. 실제로 작동한 건 두 번째 문장(= **관점**).
- 그 관점 문장이 **Axis의 존재 이유와 글자 그대로 같다.** 남이 먼저 쓰고 87K를 먹었다.
- 파트별 좋아요: 루트 283 → 2/10은 **12**. **노출은 루트가 전부 먹는다.** 뒷파트는 read-through 신호 + 실질 내용 전달용.
- 문체: 반말도 존댓말도 아닌 **음슴체("~임", "~음")**.

### 2-4. 수요의 공백
> "스레드에서 주식 추천해 준다는 거 믿고 투자해서 돈 번 사람 있나? 요즘 주식 추천글이 많이 보이네."
> — **조회 70.2K, 댓글 130**

답을 기다리는 사람이 7만 명 있는데 아무도 답을 안 놓고 있다.

### 2-5. 반증도 있음 (정보성 자체는 죽지 않는다)
- 비디펌: 건강 정보성 콘텐츠 + **댓글에 성실히 답변** → 신뢰 구축
- 잡코리아: 브랜드가 아니라 **"직장인 본캐"** 로 운영
→ 죽는 건 정보성이 아니라 **화자 없는 정보 + 답글 없는 브로드캐스트**.

### 2-6. ❌ 폐기된 가설 (영어권 SEO 블로그 출처 — 한국 스레드에 안 맞음)
| 주장 | 실제 |
|---|---|
| "100~200자가 최적, 500자 넘으면 -40%" | 좋아요 231짜리는 **긴 글**. 좋아요 3짜리는 500자 안. **길이는 변수가 아니었다** |
| "4파트 이상 타래는 도달 급감" | **10파트**로 87K |
| "링크 있으면 도달 30~50% 하락" | 루트에 **뉴스 링크 카드** 붙이고 87K |

⚠️ 이 수치들은 SNS 툴 회사 블로그가 출처다. **다시 인용하지 말 것.**

## 3. 수정 방향 (제안 — 미확정)

| # | 방향 | 비고 |
|---|---|---|
| ① | **단위: 종목 → 서사** | 종목은 서사의 *예시*로만 등장. `education.py`의 "개념 + 화제종목 실수치 삽화" 구조가 이미 이걸 하고 있었고, 그게 맞았다 |
| ② | **화자를 세운다 (음슴체)** | 반말/존댓말이 아니라 음슴체가 스레드 정보성 계정의 표준. **브랜드 결정 사항 — JEON** |
| ③ | **"추천 불신"을 정면으로 먹는다** | 지금은 프롬프트의 금지 조항일 뿐. 이걸 **콘텐츠의 주제**로 올린다 |
| ④ | **댓글이 제품이다** | 현행 `cta` 포맷(댓글로 종목 받기)이 4개 포맷 중 1개가 아니라 **엔진**이어야 함. 댓글로 들어온 종목 → Axis가 답글로 실제 분석. 알고리즘(답글 깊이) + 제품 데모를 동시에 침 |
| ⑤ | **버릴 것** | `앞으로 볼 것 ①②③` 블록(리포트 문법) / 본문 끝 링크(→ 첫 댓글) / 500자 압축 강박 |
| ⑥ | **지킬 것** | `filter_forbidden`·추천 금지 가드 / 4단계 하네스 / 결정론적 가드. **루브릭 7축만 교체** |

### 🚫 카피 원칙 (JEON 지적, 2026-07-13)
**방어적 사족·책임전가 문장 금지.** 안 하는 것 대신 **하는 것**을 긍정형으로 쓴다.
- ❌ "판단은 본인이 하는 거임" ← 독자를 향한 말이 아니라 우리를 향한 보험
- ❌ "종목 추천 안 함" ← 부정형 포지셔닝
- ✅ "좋은 숫자랑 나쁜 숫자를 같이 꺼내놓음"

면책은 프로필 bio와 서비스 내에서 이미 하고 있다. 커밋 `becc450`(방어적 사족·과장 수치 제거)에 이은 두 번째 지적.

**부수 효과**: 본문에서 "추천"이라는 단어가 사라지면 `filter_forbidden` 충돌도 함께 사라진다 → 가드에 부정문 예외를 뚫을 필요 없음.

## 5. 기술 갭 (구현 시 필요한 것)

### ✅ 이미 있는 것
- `utils/threads_client.py:41` `publish_text(text, *, reply_to_id=...)` — **답글 발행 능력은 처음부터 구현돼 있었고 아무도 안 쓰고 있다.** `:61-62`에서 컨테이너 생성 시 API로 전달.

### ❌ 없는 것

**① 파트 배열 + 순차 발행**
- 현재 `marketing_drafts`는 `text` 문자열 **하나**. 여러 파트를 담을 곳이 없음.
- `api/routes/marketing.py:401`이 `publish_text(text)`를 인자 하나로 호출하고 끝.
- 필요: 초안 스키마를 파트 배열로 확장 → 첫 글 발행 → `post_id` 수령 → 다음 파트를 `reply_to_id=직전_post_id`로 체인.
  - ⚠️ **루트 id에 전부 붙이면 타래가 아니라 평면 댓글 더미가 된다.** 반드시 직전 글에 물릴 것.
- 500자 검증(`threads_client.py:55`)을 **파트별로** 수행.
- `(n/N)` 진행 표시는 결정론적 자동 부여.

**② 부분 발행 복구 — 롤백 불가 문제** 🚨
- `threads_client.py:11` 명시: **"API 글삭제는 미지원(앱에서 수동)"**
- 5파트 중 3번째에서 실패하면 **반쪽 타래가 영구히 박힌다.** 코드로 못 지운다.
- 필요: Firestore에 `published_ids: [...]`, `published_upto: N` 기록 → 관리자 화면에서 "이어서 발행"(남은 파트만 마지막 id에 체인).
- 현행 status는 `published`/실패 이분법이라 이 상태를 표현 못 함.
- **파트 수를 3~6으로 제한하는 이유 중 하나** = 실패 지점 수를 줄이는 것.

**③ 이미지 첨부**
- `threads_client.publish_text()`는 `media_type: "TEXT"` 고정(`:60`).
- 루트에 뉴스 캡처를 붙이려면 `IMAGE` 타입 + `image_url` 필요.
- ⚠️ **Threads API는 파일 업로드를 안 받고 공개 URL만 받는다** → 이미지를 어딘가 공개 호스팅해야 함. **이 형태에서 가장 실질적인 신규 작업.**

**④ 뉴스 훅 소스**
- 이 형태는 "오늘의 사건"에서 출발. `event`/`research` 에이전트가 뉴스 원문 URL을 갖고 있는지 확인 필요.

**⑤ 레이트**
- 하루 250개 발행 한도를 파트 수만큼 소모. 현재 물량에선 여유 충분.

### 권장 순서
1. **파트 배열 + 순차 발행 + 부분성공 복구** — 첫 용처: `append_promo`를 본문 결합에서 떼어 **파트 2(첫 댓글)** 로 이동. 타래 인프라 + 링크 분리를 한 번에 얻고, 나중에 댓글 응답 루프가 그 위에 그대로 얹힌다.
2. 프롬프트/루브릭 교체 (음슴체, 서사 단위, 파트 3~6, 방어적 사족 금지)
3. 이미지 첨부 (공개 호스팅 필요)

## 6. 확정된 결정 (2026-07-14, JEON)

| # | 결정 | 내용 |
|---|---|---|
| ① | **화자 톤 = 반말/구어체** | 음슴체 아님. `@us_stock_info` 계열. ⚠️ **반말은 훅과 해석에만.** 수치는 정확하게 — "좀 많이 팔았어" ❌ / "3,200억 순매도했어" ✅. 반말이 우리 포지션과 궁합이 좋은 이유: "정답을 알려줄게"가 아니라 "같이 숫자 까보자"가 자연스러움 |
| ② | **브로드캐스트 = 서사 / 리플라이 = 종목** | 종목 분석은 *버리는 게 아니라 옮긴다.* 아무도 안 물어본 종목글 = 좋아요 3(연구원 계정이 증명). 댓글 요청에 대한 답글 = 환영받고 + 답글 깊이로 알고리즘 먹고 + 제품 데모까지 됨 |
| ③ | **현행 4포맷 폐기** | `contrarian`/`curiosity`/`technical`은 전부 "종목글 훅 변주"였으므로 ②와 함께 존재 이유 소멸. `cta` 하나만 살아남아 **엔진**이 됨 |
| ④ | **댓글 루프 = 반자동 + 검수** | 댓글 종목 감지 → Axis 분석 실행 → 답글 초안 → `/admin/marketing` 검수 큐 → JEON 승인 후 발행. 완전자동은 가드 우회 시 검수 없이 나가므로 현 볼륨에선 과설계 |
| ⑤ | **착수 = 타래 인프라부터** | 아래 §7 |

### 추가로 짚은 것 (2026-07-14)

**무사건일 커버.** 서사글은 "오늘의 사건"에서 출발하는데 휴장일·무던한 날엔 훅이 없다. → **이미 풀려 있음**: `education.py`(투자심리 함정 7 + 지표읽기 10)가 정확히 "개념 + 화제종목 실수치 삽화" 구조라 서사 소스로 그대로 쓴다. §3 ①에서 "education 구조가 맞았다"고 한 게 이 뜻.

**우리만 쓸 수 있는 서사가 아니면 무의미.** 87K 먹은 그 계정도 뉴스 하나 보고 썼다. 뉴스 훅 + 반말은 누구나 복제 가능 → 따라 하면 경쟁자 한 명 느는 것뿐. **Axis의 해자는 3파트 이후에 진짜 숫자가 나온다는 것**: 외국인 순매도 vs 기관 순매수 / 업종 전체 대비 내 종목 / PER은 싼데 매출은 3분기째 감소. 남들은 뉴스로 서사만 짜고 우리는 그 서사에 수급·섹터·밸류 **실측치**를 꽂는다.
→ **루브릭 필수 축: "데이터 파트가 실제 우리 데이터에서 나왔는가."** 지어낸 수치는 즉시 리젝(§4 주석과 동일).

## 7. 착수 1단계 — 타래 인프라 ✅ **구현 완료 (2026-07-14)**

**놀랍게도 클라이언트는 손댈 게 없었다.** `threads_client.publish_text(text, *, reply_to_id=)`가 이미 완성형이라, 바뀐 건 **초안 스키마와 발행 라우트**뿐.

> **부수 소득**: 홍보가 본문 문자열에서 빠지면서 LLM 본문 예산이 **400자 → 480자**로 돌아왔다(`LLM_BUDGET`). 홍보 블록만큼 깎아 쓰던 것을 되찾은 것 — 종목글·브리핑·교육 4개 생성기 전부에 적용된다.

### 7-1. 스키마 (`marketing_drafts/{id}`)
```
parts: list[str]          # 신규 — 진실의 원천. 1개면 단일글, 2개 이상이면 타래
text:  str                # 유지 — parts[0]. 기존 목록 UI/미리보기 하위호환
published_ids:  list[str] # 신규 — 발행된 파트의 post_id, 순서대로
published_upto: int       # 신규 — 성공한 파트 수(0-base 아님, 개수)
status: draft|approved|archived|published|partial   # partial 신규
```

### 7-2. 발행 (`POST /drafts/{id}/publish`, `marketing.py:360`)
1. **발행 전 전체 검증** — 파트별 500자, 파트 수 3~6, `filter_forbidden` 전 파트. **한 파트라도 실패하면 아무것도 발행하지 않는다**(길이 초과로 반쪽 타래가 박히는 걸 원천 차단).
2. 순차 발행: `reply_to_id = 직전 파트의 post_id`.
   - 🚨 **루트 id에 전부 붙이면 타래가 아니라 평면 댓글 더미**가 된다. 반드시 **직전 글**에 물릴 것.
3. **파트마다 즉시 Firestore 기록** (`published_ids` append, `published_upto` +1). 중간에 죽어도 상태가 남는다.
4. 전부 성공 → `status=published`. 중간 실패 → `status=partial` + 502.

### 7-3. 이어서 발행 (복구)
- API 글삭제가 **미지원**(`threads_client.py:11`)이라 반쪽 타래는 **영구히 박힌다. 롤백 불가.**
- 따라서 유일한 복구책 = **전진**. `status=partial`이면 같은 엔드포인트가 `published_upto`부터 재개하고 `published_ids[-1]`에 체인.
- 파트 수를 3~6으로 묶는 이유 중 하나 = 실패 지점 수 줄이기.

### 7-4. 첫 용처 — `append_promo` 이동
현행: 홍보 1줄을 **본문 문자열에 결합**(커밋 `27f22a1`).
변경: **파트를 하나 더 붙인다**(문자열 결합 X). 링크가 본문에서 분리돼 마지막 파트 뒤 댓글로 내려감.
→ 타래 인프라 + 링크 분리를 한 번에 얻고, 나중에 댓글 응답 루프가 이 위에 그대로 얹힌다.
⚠️ 홍보는 여전히 **LLM이 아니라 가드 이후 코드가 붙인다**(기존 원칙 유지).

### 7-5. 실제 구현 위치 (2026-07-14)
- `utils/threads_client.py` — `MAX_PARTS=6`, `PART_SEP="---"`, `split_parts`/`join_parts`/`validate_parts`/`publish_thread(on_part=)`
